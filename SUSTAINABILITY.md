# Sustainability Plan — BOSC Community Library

**Project:** BOSC Community Library
**Date:** 2026-05-11
**Maintainer:** BOSC Community
**License:** Apache License 2.0

---

## Purpose

The BOSC Community Library is designed to remain useful beyond a single assignment, grant, or maintainer. Sustainability means the project can keep improving its resources, accept public contributions, preserve trust, and support educational and public-sector use without becoming dependent on one person or one funding source.

This plan explains how the library can be maintained technically, financially, legally, and socially.

---

## Sustainability Goals

| Goal | Why It Matters |
|------|----------------|
| Keep resources free and accessible | The library serves students, educators, and public institutions |
| Maintain transparent governance | Public decision-making builds trust and accountability |
| Reduce maintainer burden | Clear workflows let new contributors help safely |
| Support localization | Communities can adapt resources into relevant languages |
| Preserve legal clarity | Apache 2.0 keeps reuse, attribution, and patent rights clear |
| Build institutional adoption | Schools and government offices need stable, documented systems |

---

## Governance Model

The project should continue using an open governance model based on GitHub Issues, Pull Requests, and documented review.

### Maintainer Responsibilities

- Review issues and pull requests within a reasonable time.
- Keep `README.md`, `CONTRIBUTING.md`, and `resources/index.md` current.
- Confirm that new resources are freely accessible and properly attributed.
- Protect the project name from misleading use.
- Encourage respectful participation under the Code of Conduct.

### Contributor Pathway

New contributors should be able to begin with low-risk tasks:

1. Fix broken links or metadata.
2. Add glossary entries or improve resource summaries.
3. Translate approved resources.
4. Add new public-domain or openly licensed resources.
5. Help review pull requests after consistent contribution.

## Funding Strategy

The library should avoid relying on a single funding source. A mixed funding model is more resilient.

| Funding Source | Use Case | Risk Control |
|----------------|----------|--------------|
| Small donations | Hosting, domain names, documentation tools | Publish basic financial summaries |
| Educational grants | Translation, accessibility, and curriculum expansion | Require open outputs under compatible licenses |
| Government partnerships | Public-sector pilots and training | Preserve public governance and Apache 2.0 licensing |
| Institutional sponsorship | Maintenance time and contributor support | Avoid sponsor control over editorial decisions |
| Volunteer contribution | Resource updates and localization | Keep tasks well-scoped and documented |

The project should not accept funding that requires paywalls, exclusive access to core resources, or restrictions that conflict with the Apache 2.0 license.

## Technical Maintenance

The project is intentionally simple: Markdown files, structured JSON, and GitHub workflows. This reduces hosting cost and makes long-term maintenance easier.

### Maintenance Tasks

| Frequency | Task |
|-----------|------|
| Weekly | Review new issues and pull requests |
| Monthly | Check internal links and update resource metadata |
| Quarterly | Review resource quality, licensing, and accessibility |
| Twice yearly | Update sustainability, governance, and legal documentation |
| Annually | Publish a project status report and roadmap |

### Data and Documentation

- Keep `resources/database.json` synchronized with `resources/index.md`.
- Use stable resource IDs in the JSON database.
- Keep resource paths relative so the library can be mirrored or hosted anywhere.
- Prefer open formats such as Markdown and JSON.

---

## Community Health

Sustainability depends on people as much as documents. The project should make contribution feel clear, fair, and safe.

Key practices:

- Welcome small improvements.
- Give specific review feedback.
- Credit contributors through commit history and release notes.
- Use the Code of Conduct consistently.
- Avoid private decision-making for public project direction.

---

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Single-maintainer dependency | Slow reviews and stalled releases | Add trusted reviewers after consistent contribution |
| Broken links | Lower trust in the library | Run monthly link checks |
| Unclear licensing | Blocks government or school adoption | Require source and license metadata for resources |
| Poor localization quality | Confusing translated materials | Use translation trackers and reviewer notes |
| Scope creep | Project becomes hard to maintain | Prioritize core educational resources first |
| Funding pressure | Mission drift or paywall risk | Reject funding that restricts public access |

---

## Success Metrics

The project should measure sustainability with simple, observable indicators:

- Number of active contributors.
- Number of published resources.
- Number of localized resources.
- Percentage of resources with complete metadata.
- Link-check pass rate.
- Average pull request review time.
- Number of schools, libraries, or institutions using the project.

## Conclusion

The BOSC Community Library can remain sustainable by staying simple, transparent, legally clear, and community-led. Its strongest long-term strategy is to combine open governance, low-cost technical infrastructure, careful licensing, and partnerships that strengthen public access instead of restricting it.
